# 发布证件照：一个关于信任的源头故事

作者：沈澜

## 开头：一个所有维护者都认识的场景

你刚修完一个 bug，跑了测试，更新了 CHANGELOG，打了 tag。项目在你的电脑上一切正常。

然后你要去告诉别人：这个版本可以用了。你去 GitHub 点开仓库，写 release note，贴链接，等别人点进来。

点进来的人会看到什么？一行 README 里写着的"跑 npx 就能用"。

他们会相信吗？

他们不会。因为"跑 npx 就能用"这句话，维护者可以写在任何项目里——不管它能不能真的装上去、装上去能不能跑、跑起来会不会当场报错。用户已经被"快速开始"伤害过太多次了。

这就是 Release Evidence Card CLI 想解决的根本问题：**你不是没有做好发布准备，你是没有一种办法让点进来的人一眼看到你真的做好了发布准备。**

---

## 冲突：测试覆盖率 ≠ 用户信任

很多维护者在发版前做的事是：

- 跑一遍测试（通过率 100%）
- 检查 CI 是否绿了
- 看一眼 README 格式对不对

这些事都有工具做。但它们全部是关于"代码是对的"，而不是关于"包是可以信任的"。

一个真实的发布场景里，用户遇到的最常见三连问是：

1. 这个包真能装上去吗？—— CI 只在远程跑，本地 npm pack 出来的东西到你电脑上会不会缺文件，没人知道。
2. 装上去之后第一条命令能跑吗？—— README 里写的 demo 命令可能早就过期了，但没人发现。
3. 项目有没有基础的治理文件？—— LICENSE 有没有？SECURITY.md 在哪里？CHANGELOG 是不是认真的？

这三件事，现有的 CI、测试框架、lint 工具一个都不回答。

---

## 转折：一张卡片能做什么

去年我在集市边看摊主怎么吸引路人，学到一件事：**一个人停下来看你，不是因为你在招牌上写了"好货"，而是因为你在他面前打开箱子，拿出东西，让他看到、摸到、试到。**

同理，在 GitHub 上让一个陌生开发者信任你的发布，不是靠 README 写得多漂亮，而是靠他看到你实际做了一遍：

- 你确实 `npm pack` 出了 tarball
- 这个 tarball 确实能在临时沙箱里干净安装
- 安装之后第一条命令确实能跑
- 项目里确实有 LICENSE、SECURITY.md、CHANGELOG.md

把这一套过程的结果压到一张可分享的 Markdown 卡片上——就是 Release Evidence Card CLI 在做的事。

它不像 CI 那样在远程跑一圈告诉你"通过"，而是在你的电脑上、用你即将发布的真实 tarball，真装一次、真跑一次，然后把所有证据写进一张卡片。任何人点开这张卡片，看到的不是"声称"，而是不可否认的事实记录。

---

## 为什么是"证件照"

何煦把它叫作"发布证件照"，这个词很准。

证件照不证明你是什么人——它只证明你在拍摄那一刻穿着那件衣服坐在那盏灯下面。它给的是"这一刻真实发生过"的背书。

同样，Release Evidence Card 不证明你的项目没有 bug、没有漏洞、架构优雅——它只证明：

- 你确实打了一个 tarball
- 这个 tarball 确实能装
- 第一条命令确实能跑
- 治理文件确实存在

这四条，正是用户点进一个 GitHub 项目，在决定 star 或 clone 之前最想确认的事。

---

## 五分钟原则

项目有一个硬约束：**从 npx 到拿到证据卡，不超过五分钟。**

这个约束的来历很实际。用户决定试用一个 CLI 工具的注意力窗口就在五分钟以内。超过五分钟他们就会觉得"算了先放着吧"——然后永远不会回来。

所以整个工具的设计都是围绕这个时间窗口展开的：

- 不需要注册账号
- 不需要 clone 项目
- 不需要写配置文件
- 直接 `npx release-evidence-card check`，五分钟后拿到卡片

---

## 它不是什么

一件事值不值得信任，有时候不取决于它能做什么，而取决于它承认自己不能做什么。

Release Evidence Card CLI 有明确的边界：

1. **它不替代你的单元测试和 CI。** 它是发版前最后那一步"开箱验货"，不是质量保证体系。
2. **它不能证明项目没有漏洞。** 它只能证明 tarball 安装和首条命令通过。安全审计是另一件事。
3. **它不上传任何产物到第三方。** 所有证据本地生成，不会把你的 tarball 发到某个服务器上。

---

## 给维护者的话

如果你已经在维护一个 npm CLI 项目，以下是你应该试试这个工具的信号：

- 你发版前要手动检查一堆东西，但每次都怀疑自己漏了什么
- 你的 README 里写着"npx xxx"快速开始，但你其实不确定那行命令在别人的电脑上能不能跑通
- 你的项目治理文件齐全，但从来没有一个地方把它们集中展示出来
- 你想要一个新的 star，但不想靠"给个 star 吧"去要

如果你中了任何一条，在自己的项目目录里跑一次：

```bash
npx release-evidence-card check
```

五分钟后你会拿到一张卡片。那张卡片不一定证明你的项目完美——但它能证明你不是随便发个版本就走的人。
